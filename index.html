The Problem with On-Chain Randomness

Most games ask you
to trust the hash.

On Solana, "random" outcomes derived from block hashes can be manipulated by validators. Here's the difference between trusting a promise and verifying a proof.

⚠️
Hash Randomness (Other Games)

Block hash-based randomness lets validators who produce blocks add noise to influence outcomes. Slot leaders can withhold blocks until the "random" result is in their favour. MEV tools like Jito make this attack more accessible than ever.

// ❌ Vulnerable — validator can manipulate
let random = recent_blockhash[0]
// slot leader controls this hash
✅
Magicblock VRF (TIPF)

Verifiable Random Function generates a cryptographic proof using Curve25519/RFC 9381. The proof is verified on-chain before the value is used. No validator, no operator — nobody can predict or influence the outcome.

// ✅ Trustless — anyone can verify
vrf.requestRandomness(round_id)
// proof: Curve25519 / RFC 9381
→ on-chain verified before use
How VRF Works

4 steps. Zero trust required.

The Magicblock VRF process runs entirely within Ephemeral Rollups — free, sub-50ms, and audited by Zenith.

01
TIPF program requests randomness
At the end of each mining round, the TIPF smart contract submits a randomness request to the Magicblock VRF oracle queue via CPI (Cross-Program Invocation).
requestRandomness(round_id)
02
Oracle computes value + cryptographic proof
An independent oracle picks up the request and generates a random number alongside a cryptographic proof using the Curve25519 elliptic curve (RFC 9381 standard). The proof mathematically ties the result to the request — impossible to forge.
Curve25519 / RFC 9381 / SHA-512
03
Proof verified on-chain before use
The proof is submitted on-chain and verified by the Magicblock VRF program before the random value is ever used. If it fails verification, the value is rejected. This is the key step that makes it trustless — anyone can audit it.
On-chain verification → callback
04
Winner selected. Round settled. Verifiable forever.
TIPF uses the verified random bytes to select the winning block. The proof lives on-chain permanently — any third party can verify that the draw was fair, after the fact, at any time. VRF: audited by Zenith. 1 transaction vs 50–100 for older VRF systems.
Sub-50ms latency · Free on Ephemeral Rollups

Tokenomics

The deflationary flywheel.

Every mining round feeds a self-reinforcing economic engine. More mining → more burns → less supply → price support.

⛏
Miners stake SOL in blocks
Each round, miners deploy SOL across 25 blocks in either the Aggressive or Stable pool. The protocol charges a 1% operation fee to cover VRF costs and RPC expenses.
🏆
Round settles — 10% protocol fee taken
From the winning block's SOL payout, 10% is taken as a protocol fee. This entire amount is used to buy back TIPF tokens from the open market via Jupiter Aggregator.
🔥
90% of buyback permanently burned
90% of repurchased TIPF is permanently removed from the total supply. Forever. No re-minting. No recovery. This creates a continuous deflationary pressure on every single mining round.
💎
10% distributed to nodes & stakers
The remaining 10% of buyback rewards permanent stakers and node holders. More mining activity → more revenue → more rewards for long-term aligned holders.
♻️
Reduced supply → increased scarcity → price support
As supply contracts with each burn, and TIPF can only be acquired through mining (Token2022 hook enforces this), organic scarcity builds — feeding back into miner incentives to mine more.
🔗
Token2022 Transfer Hook: Supply Control at the Contract Level
TIPF uses Solana's Token2022 program with a custom transfer hook — meaning the rules aren't just policy, they're enforced in code. TIPF tokens can only be acquired through mining. They can only be sold on approved DEXs. C2C trading happens only via OTC. No centralized mint. No team allocations through secondary sales. The supply curve is determined entirely by mining activity.
